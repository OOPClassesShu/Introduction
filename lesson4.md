# Правило пяти в C++

## Проблема управления ресурсами

C++ предоставляет программисту прямой контроль над памятью и другими ресурсами (файлы, сокеты, мьютексы и т.д.). \
Если класс владеет ресурсом (например, выделенной динамической памятью), он должен отвечать за его освобождение. \
Без специальных механизмов копирование таких объектов может привести к ошибкам: двойному освобождению, утечкам памяти, неопределённому поведению.

## Правило трёх

До C++11, если класс управлял ресурсом, который требовал нетривиального уничтожения, обычно требовалось определить три специальные функции-члена:
- Деструктор (~T()) – освобождает ресурс.
- Конструктор копирования (T(const T&)) – создаёт копию ресурса (глубокое копирование).
- Оператор присваивания копированием (T& operator=(const T&)) – освобождает старый ресурс и копирует новый.

Почему это правило возникает? \
Если компилятор генерирует эти функции автоматически, он выполняет почленное копирование полей. \
Для указателей это означает просто копирование адреса, а не копирование данных. \
В результате два объекта будут указывать на одну и ту же память, и при уничтожении одного из них произойдёт \
двойное освобождение (если оба вызовут delete), либо утечка, если память не будет освобождена вовсе.

## Правило пяти

С появлением семантики перемещения в C++11 добавились ещё две специальные функции: 
- Конструктор перемещения (T(T&&)) – перехватывает ресурс у временного объекта.
- Оператор присваивания перемещением (T& operator=(T&&)) – перемещает ресурс из одного объекта в другой.
- 
Правило пяти гласит: если класс управляет ресурсом, то обычно требуется определить все пять (или явно запретить некоторые из них). \
Часто при этом реализуют копирующие операции через глубокое копирование, а перемещающие – через перехват ресурса (обнуление источника).

```
#include <iostream>
#include <cstring>

class String {
    char* data;
    size_t len;

public:
    // Конструктор
    String(const char* str = "");

    // Деструктор
    ~String();

    // Конструктор копирования (глубокое копирование)
    String(const String& other);

    // Оператор присваивания копированием (copy assignment)
    String& operator=(const String& other);

    // Конструктор перемещения
    String(String&& other) noexcept;

    // Оператор присваивания перемещением
    String& operator=(String&& other) noexcept ;
    
};

int main() {
    String s1("Hello");
    String s2 = s1;           // копирование
    String s3("World");
    s3 = s1;                  // копирующее присваивание
    String s4 = std::move(s1); // перемещение (s1 становится пустым)
    s3 = std::move(s4);       // перемещающее присваивание
    // s1, s2, s3, s4 уничтожатся автоматически
}
```
## Правило нуля

Современный C++ рекомендует следовать правилу нуля: не писать собственные деструкторы, конструкторы копирования/перемещения и операторы присваивания, \
если класс не управляет ресурсами напрямую. Вместо этого используйте готовые обёртки, которые уже реализуют правильное поведение: \
std::vector, std::string, std::unique_ptr, std::shared_ptr и т.д. \
Правило нуля и правило пяти не противоречат друг другу: если вы вынуждены управлять ресурсом (например, из-за интерфейса на C), реализуйте все пять. \
Если же можно обойтись без этого – не пишите их вручную.
